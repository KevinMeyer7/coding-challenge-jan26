Key Design Decisions

  1. SurrealDB as the primary data store. SCHEMALESS tables — flexible for nullable/optional fruit attributes. HTTP API used directly from Deno (zero SDK dependencies).
  2. Three separate tables: apple, orange, match. Type-safe queries, independent indexes, clear relationship modeling.
  3. Edge Functions as stateless backend. Each function follows the prescribed 5-step flow. All state in SurrealDB.
  4. Client-side rendered dashboard. Interactive chat UI + real-time updates are better as client components.

2 Matching Algorithm

Bidirectional Compatibility Scoring

  • Forward score: How well does the candidate satisfy the incoming fruit's preferences?
  • Reverse score: How well does the incoming fruit satisfy the candidate's preferences?
  • Mutual score: √(forward × reverse) — geometric mean of both
Why geometric mean? It penalizes imbalance. A 100% + 0% pair scores 0% (geometric) vs 50% (arithmetic). Both sides must be satisfied — this mirrors real matchmaking.

Scoring Rules

TypeMethod
BooleanExact match = 1.0, mismatch = 0.0
Numeric rangeIn range = 1.0, out of range = exponential decay
Shine factorExact = 1.0, partial credit by scale proximity
Null attribute0.5 (neutral)
Omitted preferenceNot counted (any value OK)

Exponential Decay for Numeric Ranges

Size 8.1 when max preference is 8.0 shouldn't score 0. e^(-distance/scale) provides graceful degradation near boundaries, rapid falloff for large deviations.

3 LLM Integration

Optional and gracefully degradable:

  1. OPENAI_API_KEY set → calls OpenAI chat completions
  2. API fails → template-based narrative fallback
  3. No key → template generator directly

Prompt creates a "Perfect Pear" matchmaking show host. Provides fruit descriptions + top 3 scores. Output capped at 200 words.

Why Chat UI?

  1. Natural mapping: System generates language — chat is most intuitive
  2. Sequential flow: arrive → introduce → search → match maps to timeline
  3. Engaging: 600ms message animations create narrative feel
  4. Familiar: Universally understood interface pattern

5 Data Flow

When user clicks "New Apple":

  1. Frontend POSTs to edge function
  2. Generate random apple (Box-Muller distribution)
  3. Generate natural language descriptions
  4. Store apple in SurrealDB
  5. Fetch all oranges, score each (forward + reverse + mutual)
  6. Select top 3, generate LLM narrative
  7. Store best match, return response
  8. Chat messages animated in, stats refreshed

7 Trade-offs

With More Time

  1. Trigger.dev: Background batch matching jobs
  2. Effect library: Deeper edge function error pipeline
  3. Network graph: D3.js force-directed visualization
  4. Realtime: Supabase Realtime for cross-tab updates
  5. A/B testing: Compare scoring algorithms
