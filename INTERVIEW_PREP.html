Fruit Matchmaking System — Technical Review

Kevin Meyer | Clera Coding Challenge | April 2026

1 System Architecture

Three-tier architecture: Next.js 16 frontend on Vercel → Supabase Edge Functions (Deno) → SurrealDB Cloud. LLM narratives via GPT-4o-mini through the Vercel AI SDK. State managed with Zustand, error handling with Effect, analytics with Recharts.

Three edge functions: get-incoming-apple, get-incoming-orange (5-step matching flow), and get-stats (dashboard data). All deployed to Supabase Cloud. SurrealDB accessed via HTTP API with safe parameterized queries.

2 Matching Algorithm — 6 Design Decisions

Decision 1: Bidirectional Scoring

Most naive systems check "does A like B?" We check both directions. Forward: does the candidate satisfy our preferences? Reverse: do we satisfy the candidate's preferences? A match is only good if both sides are happy.

Decision 2: Geometric Mean for Mutual Score

Mutual = √(Forward × Reverse). Arithmetic mean gives 50% to a 100%/0% pair — hiding a one-sided mismatch. Geometric mean gives 0% — correctly penalizing imbalance. Same principle as F1 score in information retrieval.

Decision 3: Weighted Criteria (Dealbreakers vs. Cosmetics)

CriterionWeightRationale
Worm3.0xDealbreaker — nobody wants a worm
Chemicals2.5xStrong health concern
Size, Weight1.0xStandard physical preferences
Stem0.8xMild preference
Leaf, Shine0.7xCosmetic / nice-to-have

A worm mismatch tanks the score 3x harder than a shine mismatch. Models real-world preference intensity.

Decision 4: Exponential Decay for Numeric Ranges

If preferred size is 5-10 and candidate is 10.5, score is e^(-0.5/scale) ≈ 85%, not 0%. Graceful degradation near boundaries, harsh penalty far away. Scale derived from range span.

Decision 5: Ordinal Distance for Shine Factor

Scale: dull → neutral → shiny → extraShiny. "Shiny" vs "extraShiny" gets partial credit (1 step). "Shiny" vs "dull" gets almost none (2 steps). Treats the attribute as ordinal, not nominal.

Decision 6: Null = 0.5 (Maximum Entropy)

Unknown attributes score 0.5 — the information-theoretic neutral point. Missing data shouldn't punish or reward. Omitted preferences are not scored at all (any value acceptable).

3 LLM Integration

Two paths: Edge functions call OpenAI directly for the initial narrative. Frontend has an AI SDK route (/api/chat) using the Vercel AI SDK. Both use GPT-4o-mini. Template fallback if API is unavailable — system never breaks.

Prompt design: "Perfect Pear" matchmaking show host persona. Receives fruit attributes + top 3 match scores. Generates unique, witty narrative per match with fruit puns and score explanations.

4 Frontend Design Decisions

Chat UI as the Visualization Medium

The system generates natural language — a chat interface is the most intuitive display. Three-phase flow: fruit introduces itself (with visual profile card) → system searches pool → best match revealed with score breakdown and narrative.

Bidirectional Scatter Plot

Forward score (x) vs Reverse score (y). Diagonal = perfect balance. Makes the geometric mean algorithm visible. Dots above diagonal = "they like us more." Below = "we like them more." This is the visualization that demonstrates algorithmic understanding.

Insight-Driven Metrics (not raw numbers)

"Pool Health" (% matched) tells more than "Matches: 25". "Balance" (forward vs reverse avg) shows if the algorithm favors one direction. "Best Match" highlights system capability. Hover hints explain each metric.

5 Data Flow (End-to-End)

  1. User clicks "New Apple" → Zustand dispatches action
  2. Effect pipeline: fetch with retry (2x exponential backoff) → Supabase Edge Function
  3. Edge function generates random apple (Box-Muller distribution) with natural language descriptions
  4. Apple stored in SurrealDB Cloud (HTTP API, safe parameterized queries)
  5. All oranges fetched → weighted bidirectional scoring against each candidate
  6. Top 3 matches selected by mutual score → GPT-4o-mini narrative generated
  7. Best match stored in SurrealDB → full response returned to frontend
  8. Chat messages animate in (Framer Motion, 500ms stagger) → stats auto-refresh

7 Questions I'd Ask

  1. How would match quality be measured in production? Is there a human feedback loop?
  2. Would you want synchronous matching or Trigger.dev batch processing?
  3. Expected data volume — hundreds or millions of fruits?
  4. Would you explore SurrealDB graph features (RELATE) for match relationships?
  5. How should the LLM integration scale — dedicated model or per-match API calls?
